# 【合意用】P2：RACI定義・運用ルール集 v1.0

**目的:**
本資料は、P2「組織基盤強化」プロジェクト におけるRACIチャート(ID 2.8) の作成にあたり、R/A/C/Iの「共通言語（定義）」と「運用ルール」を事前に合意し、ワークショップ(ID 2.8)での議論の混乱を防ぎ、憲章の課題を確実に解消することを目的とします。

---

## 1. 「RACI」の共通言語（定義）

| 記号 | 名称 | 定義（本プロジェクトにおける解釈） |
| :--- | :--- | :--- |
| **R** | **Responsible**<br>（実行担当者） | ・その業務・タスクを、**実際に手を動かして実行する**人（または役割）。<br>・複数名（複数役割）がRになることは許容される。 |
| **A** | **Accountable**<br>（実行責任者） | ・その業務・タスクの**最終的な結果**に対して**責任（説明責任）**を負う、**ただ一人の**人物（または役割）。<br>・R（実行担当者）を兼務することもある。<br>・**最終的な意思決定権**を持つ。 |
| **C** | **Consulted**<br>（協業・相談先） | ・実行前に、**必ず相談・協議**し、その意見を反映する必要がある人物（または役割）。<br>・コミュニケーションは**「双方向」**である。 |
| **I** | **Informed**<br>（報告・情報共有先） | ・実行後（または実行中）に、進捗や結果を**報告・共有**する必要がある人物（または役割）。<br>・コミュニケーションは**「一方向」**（報告のみ）である。 |

---

## 2. RACIの「絶対ルール」（憲章課題の解決）

ワークショップ(ID 2.8)では、以下のルールを遵守し、憲章の課題を解決します。

* **ルール1: 「A」は必ず1名、かつ1名のみ。**
    * `A`が複数いる行 ＝ 憲章課題**「役割の重複」** が発生。WSの場で、`A`を1名に決定します。
    * `A`が0名の行 ＝ 憲章課題**「業務の隙間」**（＝誰も責任を取らない業務）。WSの場で、`A`を1名アサインします。

* **ルール2: 「R」は最低1名。**
    * `R`が0名の行 ＝ 憲章課題**「業務の隙間」**（＝実行する人がいない）。WSの場で、`R`を最低1名アサインします。

* **ルール3: 「R」が多すぎないか？**
    * `R`が不必要に多い行 ＝ 憲章課題**「リソースの無駄」**（＝同じ作業に過剰な人員が関与）。WSの場で、本当にRが必要か（`C`や`I`で十分ではないか）を議論します。

* **ルール4: 「内部」と「外部」の連携を明確にする。**
    * 憲章課題である「内部と外部の連携ルール」を解決するため、特に「外部（SFFなど）」が関わる行（タスク）については、`R` `A` `C` `I` を明確に定義します。
