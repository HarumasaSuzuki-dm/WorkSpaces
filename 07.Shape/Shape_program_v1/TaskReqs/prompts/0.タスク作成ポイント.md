トップティアのコンサルティングファーム（マッキンゼー、BCG、あるいはAppleやGoogleのPM組織など）において、タスク依頼ドキュメントは単なる「作業指示書」ではありません。それは **「期待値の完全な合意形成（Alignment）」** であり、 **「自走を促すための地図」** です。

クライアント自身が実務を行う場合、曖昧さは「手戻り」だけでなく「信頼の毀損」に直結します。

ご要望の **「意図の正確な理解」「自発的なアクション」「ゴールへのアライン」** を実現するために、Notionドキュメントに記載すべき要素をMECE（漏れなくダブりなく）に定義しました。

---

### コンサル兼PMのためのタスク定義フレームワーク（MECE）

タスク定義を構成する要素を以下の5つの次元に分解します。

1.  **Why (背景と目的)**：なぜやるのか、上位戦略との接続
2.  **What (成果物定義)**：何を作るのか、完了条件
3.  **How (実行指針)**：制約条件、リソース、進め方
4.  **When (時間軸)**：納期、マイルストーン
5.  **Next (接続)**：完了後の世界、次のアクション

これらをNotionのプロパティや本文に落とし込むための具体的な項目は以下の通りです。

#### 1. Context & Alignment (文脈と戦略的整合性)
クライアントに「やらされている感」を与えず、「事業成長のために必要だ」と認識させるパートです。
* **Background / Issue (背景・課題)**
    * 現在どのような課題があり、なぜ今このタスクが発生したのか。
* **Objective (このタスクの目的)**
    * このタスクを完了することで、誰の何が解決されるのか。
* **Strategic Impact (上位目的への貢献)**
    * プロジェクト全体のKGI/KPIに対して、このタスクがどう寄与するか（＝やる意義の明確化）。

#### 2. Output Definition (成果物の定義) - 最重要
ここが曖昧だと、期待と異なるアウトプットが出てきます。
* **Deliverables (具体的な成果物)**
    * 形式（Excel, PPT, Notion, デザインデータなど）
    * 数量（3案出すのか、1つの決定稿か）
* **Definition of Done (DoD / 完了の定義)**
    * どの状態になれば「完了」と言えるか。（例：「ドラフト作成」ではなく「マネージャーの承認完了」まで含むか）
* **Quality Criteria (品質基準・成功要件)**
    * 求められる精度（「ブレストレベルの箇条書き」か「顧客提出レベルの完成度」か）。
    * 判断軸（スピード優先か、正確性優先か）。

#### 3. Constraints & Resources (制約とリソース)
自走を促すために「ガードレール（やってはいけないこと）」と「武器（参考情報）」を渡します。
* **Scope / Out of Scope (スコープの範囲・対象外)**
    * やるべき範囲と、今回は「やらなくていいこと（捨てていいこと）」の明記。これにより迷いを消します。
* **References & Assets (参考資料・過去事例)**
    * 参考になる過去のドキュメント、議事録、テンプレートへのリンク。
* **Stakeholders (関係者・承認者)**
    * 誰にヒアリングすべきか、誰の承認が必要か。

#### 4. Timeline & Checkpoints (時間軸とマイルストーン)
締め切り直前のサプライズを防ぎます。
* **Final Deadline (最終納期)**
* **Checkpoints (中間レビューのタイミング)**
    * 「着手後10%（方向性確認）」と「進捗80%（仕上げ前確認）」のタイミング設定。

#### 5. Next Actions (次への接続)
「作って終わり」にさせないための仕掛けです。
* **Usage (成果物の使い道)**
    * この成果物は、次のどの会議で使われるか、誰が見るか。
* **Suggested Next Step (完了後のネクストアクション想定)**
    * このタスクが終わったら、自動的に次は何がタスク化されるはずか（ここを意識させると先読みして動いてくれます）。

### 📝 Notion実装用テンプレート（コピー＆ペースト用）

この構成をNotionのページテンプレートとして登録し、タスク起票時に毎回呼び出すことを推奨します。
```
## 1. Why: 背景と目的
* **背景**: [なぜこのタスクが必要か、現状の課題]
* **目的**: [このタスクを通じて達成したい状態]
* **インパクト**: [プロジェクトゴール（KGI/KPI）への貢献]

## 2. What: 成果物と完了定義 (DoD)
* **提出形式**: [例: Googleスライド 5枚程度 / Excel試算表]
* **完了の定義**: [例: ●●さんのレビュー完了 / クライアントへの送付完了]
* **品質期待値**:
    * [ ] スピード優先（60%の完成度でOK） / クオリティ優先（ミス許容ゼロ）
    * [ ] 論点はMECEであること / ファクトベースであること

## 3. How: スコープとリソース
* **スコープ内**: [今回実施すること]
* **スコープ外**: [今回やらなくて良いこと・後回しにすること]

## 4. When: スケジュール
* **最終納期**: 202X/XX/XX
* **推奨チェックポイント**:
    * [ ] 骨子レビュー (XX/XX)
    * [ ] ドラフト版レビュー (XX/XX)

## 5. Next: 次のアクション
* このタスク完了後、[次の工程/会議] に接続されます。
* 完了時に意識すべき次の課題: [例: 運用体制の検討]
```

### アドバイス

クライアントにタスクを依頼する際、特に以下の2点を強調すると劇的に動きが変わります。

1.  **「Scope Out（やらないこと）」を明記する**
    * クライアント（特に真面目な担当者）は、完璧を目指して時間を使いすぎる傾向があります。「今回はデザインの調整は不要です」「データは過去3ヶ月分だけで十分です」と**捨てる勇気**を与えることが、PMとしての最大の価値提供になります。
2.  **「10%レビュー」の文化を作る**
    * Notion上で「着手して骨子ができたら、メンション飛ばしてください」と依頼します。方向性が合っているか早期に確認することで、手戻りを防ぎ、クライアントの心理的負担を下げることができます。
