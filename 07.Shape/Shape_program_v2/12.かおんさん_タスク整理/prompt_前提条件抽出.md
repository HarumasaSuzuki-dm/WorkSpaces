## 前提条件抽出の目的

1. **タスク分解の妥当性評価**：現在のタスクの切り方が適切かを判断するための材料
2. **期限設定の根拠確保**：各タスクの期限を設定する際の根拠となる情報
3. **メンバーへの説明材料**：なぜこのタスク構成・期限なのかをメンバーに説明するための材料

---

## 前提条件の抽出観点（4つのカテゴリ）

### 1. プロジェクト特性
- ゴール定義（最終成果物、成功基準）
- 制約条件（絶対期限、予算、フォーマット指定）
- 依存関係（外部依存、タスク間・成果物間の前後関係）

### 2. リソース
- 人的リソース（アサイン可能者、稼働可能時間、スキル・経験）
- 物的・情報リソース（必要な素材・データ・ツール・権限・過去実績）

### 3. タスク特性
- 作業量の見積もり根拠（過去実績、類似作業の有無）
- 不確実性（仕様変更リスク、承認待ち、外部返答待ち）
- 完了条件（何をもって完了とするか）

### 4. コミュニケーション
- 確認・承認フロー（誰の承認が必要か、決裁者）
- 情報共有（誰に何を共有するか、報告頻度・方法）

## アウトプット形式

以下のCSV形式で出力してください。

```csv
ID,カテゴリ,状態,タイトル,内容,メモ
```

### カラム定義

| カラム | 説明 | 入力ルール |
|--------|------|------------|
| ID | 前提条件の識別子 | P001, P002, ... の連番形式 |
| カテゴリ | 前提条件の分類 | 「プロジェクト特性」「リソース」「タスク特性」「コミュニケーション」のいずれか |
| 状態 | 確定状況 | 「未決事項」または「決定事項」 |
| タイトル | 前提条件の名称 | 20文字以内で簡潔に |
| 内容 | 詳細説明 | 前提条件の具体的な内容。どの領域・タスクに関係するかも記載 |
| メモ | 補足情報 | 確認先、想定選択肢、リスク、関連タスク名など |

---

## 出力ルール

1. **タスク一覧から読み取れる情報**は「決定事項」、**不明・要確認の情報**は「未決事項」とする

2. **特定タスクに関わる前提条件**は、内容またはメモ欄に関連するタスク名・領域を記載する

3. **複数タスクに共通する前提条件**は、内容欄に「共通」または「全体」と明記し、どの範囲に影響するかを記載する

4. **期限設定に影響が大きい未決事項**には、メモ欄に「★要優先確認」と記載する

5. **タスク分解の妥当性に関わる前提条件**（例：成果物の定義、完了条件、依存関係）は優先的に抽出する

6. **重複を避ける**：同一内容の前提条件は統合し、関連する領域・タスクはメモ欄にまとめて記載する

## 注意事項

- タスク一覧のカテゴリ分け（審査員、MC・実況、Shaperなど）は参考情報として扱い、前提条件は4カテゴリ（プロジェクト特性、リソース、タスク特性、コミュニケーション）で整理してください
- 「未定」「○/○」など未確定の記載は、未決事項として抽出してください
- タスク一覧に明示されていなくても、タスク分解・期限設定に本質的に必要な前提条件は積極的に抽出してください
