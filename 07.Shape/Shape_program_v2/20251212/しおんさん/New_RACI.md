## 0) コンテキスト要約（ソースからの事実抽出）

* SFF運営RACIは **Phase1（設計確定）〜Phase4（事後対応）** に分かれている。 
* RACIの目的として、**統括（しおんさん）の判断疲れを減らす／実行側の権限境界を明確にする／回せる道具にする** が明記されている。 
* 現状、しおんさんに **意思決定・進捗・品質** が集中し、**C（相談）とA（最終責任）の重複** により心理負荷・判断疲れが発生している。 
* 実行者R側（神保さん）に **期日遵守/タスク漏れが頻発** し、それが委譲を阻害し **マイクロマネジメントの悪循環** を生んでいる。 
* **指揮命令系統が二重化**（しおん→町田 直指示 / 神保経由が混在）し、**神保→町田への移譲が滞留** している。 
* コミュニケーション特性（FFS等）の違いで、特に **しおん×神保の対極** では噛み合わず、**通訳（翻訳/通訳）** が必要になる局面がある。 
* **数値分析・管理（SNSインサイト/チケット販売など）** の役割が不在で、数字で改善を回せていない。 
* memo.mdにある4打ち手（棚卸し、PMO意義、タスク翻訳15分MTG、体制変更フィードバック）は **すでに検討・実装フェーズ** として目的/課題/論点まで構造化されている。

## 1) 事実根拠（“判断停止・手戻り・二重指揮・欠落機能”の根拠）

* **判断停止/判断疲れ**：CとAが重複し「相談役なのに最終責任者」状態、進捗/品質まで心理的に背負ってしまう。 
* **委譲不全**：神保さんの期日遵守・漏れが頻発→任せづらく、しおんさんが過度確認に。 
* **二重指揮**：神保→町田の移譲が進まず、しおん→町田直指示と神保経由が混在。 
* **翻訳不足による手戻り**：「よしなに」だと期待値に届かず修正コストが増える、翻訳工数を確保する仕組みがない。 
* **欠落機能（数字で回す）**：SNS/チケット等の数字を見て分析し改善策を出す役割がいない。 
* **個別対応の名残**：請求書送付・振込などがしおんさん宛/しおんさんが処理している。

## 2) 問題構造（原因→症状→影響→根本原因）

```
原因（構造）
- A/C境界が曖昧（不可逆と日常判断の区別が弱い）
- 入口（指示/依頼）が複数（PMO不在または弱い）
- タスク定義が曖昧（成果物/期限/品質が翻訳されない）
- 実行の信頼（期日/漏れ）の不足
- 数値モニタリングがない（早期アラート不在）

症状
- しおん/統括の判断疲れ・判断停止
- マイクロマネジメント化
- 二重指揮・移譲滞留
- 手戻り増・進捗不透明

影響
- 停止/遅延/炎上リスク増
- 手戻りコスト増
- 属人化が進み、毎年同じ問題が再発

根本原因（最短で直すべき“テコ”）
- 「入口の一元化」＋「判断境界線（閾値）の明文化」＋「翻訳MTGの定例化」＋「数字アラート」
```
