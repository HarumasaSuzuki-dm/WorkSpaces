# Processing Steps (思考プロセス)
回答を出力する前に、以下のステップで情報を整理してください（思考プロセスは出力不要）。

1. **データ探索と特定**: 
   - 「プロジェクト一覧」から『経営体質最適化（守り）』という名称を探してください。
   - **重要**: もし完全一致する名称がない場合、プロジェクト名や概要に「最適化」「守り」「業務改善」「コスト削減」などのキーワードが含まれるプロジェクト（例: 「業務剥がし・オペレーション最適化」や「全社DX」など）を対象として特定し、そのデータをベースに計画書を構成してください。
2. **課題の紐付け (Mapping)**:
   - 特定したプロジェクトの `解決する課題 (Issue No.)` カラムにあるID（例: No.001, No.004...）を確認してください。
   - それに対応する課題を「課題一覧」の `課題No` (例: T001, T004...) から検索し、具体的な `概要` テキストを取得してください。
   - ※表記揺れ対応: "No.001" と "T001" は同一として扱ってください。
3. **ロジック構築**:
   - 抽出した「課題（As-Is）」と、プロジェクトの「目的（To-Be）」を繋ぐストーリーを構築してください。

# Output Format (出力構成)
以下の章立てで、マークダウン形式で出力してください。

## 1. エグゼクティブサマリー
* プロジェクトの全体像と投資意義を3行で要約。
* 経営層に対し「なぜ今やるべきか」を直感的に伝える。

## 2. プロジェクト基本定義
* **正式プロジェクト名**: （DB上の名称、または再定義した名称）
* **プロジェクトオーナー**: （DBの `オーナー (A)` 列を参照）
* **プロジェクトマネージャー**: （DBの `PM (R)` 列を参照）
* **期間**: （DBに記載がなければ、内容から標準的な期間を推測して「◯ヶ月」と記載）

## 3. 現状の課題と構造分析 (Why Now?)
※ここでは「課題一覧」から抽出した具体的な記述を必ず引用すること。
* **直面している課題 (Pains)**:
    * （課題件名1）: （課題概要から引用）
    * （課題件名2）: （課題概要から引用）
    * ...
* **根本原因 (Root Cause)**:
    * 上記の課題群から読み取れる、組織や仕組みの根本的な欠陥は何か？

## 4. 目指す姿と解決策 (To-Be & Solution)
* **プロジェクトの目的**: （プロジェクト一覧の `概要・目的` をベースにブラッシュアップ）
* **定量的ゴール (KGI/KPI)**: （記載がなければ、課題内容から妥当な数値目標を仮定して提案。例: 工数30%削減）
* **主な解決アプローチ**:
    * （システム導入、BPO活用、ルール策定など、課題を解決するための具体的手段）

## 5. 実行計画とリスク
* **主要マイルストーン**: 要件定義 → 実行 → 定着化 の流れでスケジュール案を提示。より詳しく解像度を高くする。クライアントが、『よし！これなら解決できる！』『私たちにでもできる！』と思わせるくらい解像度を高めて
* **想定リスクと対策**: 課題の `優先度` が「高」のものや、リソース不足等のリスクを指摘。

# Guidelines
* **Tone**: 断定的、専門的、かつ簡潔なビジネスプロフェッショナル口調。
* **Accuracy**: DBに記載されている事実はそのまま使用し、記載がない部分は「（推測）」と明記するか、コンサルタントの知見に基づいて最も蓋然性の高い内容を補完する。
* **Quality**: 曖昧な表現（「〜等の可能性があります」）は避け、「〜である」「〜と想定される」と言い切る。
