# 【最終確定版】プロジェクト支援方針：理念と実務の「接続」と「整流」

「理念先行型COO」である中井様の想いや熱量を、鈴木様が「論理・数値・構造」で補完し、組織全体が迷わず走れる状態を作ります。特に**「SFF事業」「コミュニティ」「法人研修」**という個別の事業が、全体としてどう繋がり、どの財布（PL）で回るのかを可視化します。

## 1. 支援の中核テーマ：「点」を「線」にする

これまでの課題（PL、組織、IT、ロードマップ）を個別のタスクとして処理するのではなく、全てが**「どう紐づいて、どのような優先順位で動くべきか」**という全体像の接続を行います。

*   **Vision Logic（ビジョンと数字の接続）**
    *   **課題:** なーすけCEOの掲げる「10年後のビジョン（理想）」と現場の「3年後の現実」の間に乖離がある。特にSFF事業の価格設定や投資判断において、**「PL（損益計算書）」という判断基準が機能しておらず**、感覚的な意思決定になっている。
    *   **支援:** 鈴木様がPL作成・運用を巻き取り、「これをやれば達成できる」と信じられる根拠（数字・リソース計算）を付与したロードマップを作成する。
*   **Mental Sorting（思考の棚卸しと選別）**
    *   **課題:** 目の前のミクロな課題（誰がタスクを拾うか）から未来の話（2026年の展開）までが混在し、何に集中すれば「大丈夫」なのかが見えていない。
    *   **支援:** 中井様が抱えるモヤモヤを全て吐き出す場を設け、「これは今やるべき」「これは一旦放置でOK」という判断・交通整理を行う。

## 2. ロードマップの「正体」と構成要素

クライアントがロードマップ作成において「何が必要か不明確」と感じている現状を踏まえ、全員が腹落ちして推進するための「ロードマップの定義」を以下のように設定します。ロードマップとは単なる日程表ではなく、**「完了条件（成果物）の合意」**です。

| 構成要素 | 定義（Why） | 問い（Check） |
| :--- | :--- | :--- |
| **1. 成果物**<br>(Deliverables) | 各フェーズで**「何ができあがればゴール（完了）とするか」**という具体的なアウトプット。これがないと「やったつもり」で終わる。 | 「これさえあれば、次の判断ができるか？」 |
| **2. 依存関係**<br>(Logic) | **「Aが終わらないとBは始められない」**という論理的な順序。ここを無視すると手戻りやボトルネックが発生する。 | 「なぜ、今これをやる必要があるのか？」 |
| **3. マイルストーン**<br>(Checkpoints) | 意思決定を行い、次のフェーズに進むための**「関所」**。ここで承認（Go/No-Go）を行う。 | 「いつまでに、誰が承認するのか？」 |
| **4. リソース**<br>(Resources) | 実行するために必要な**「人の稼働」**と**「資金」**。 | 「絵に描いた餅になっていないか？」 |

**★重要な合意事項:**
スケジュール（日付）はあくまで結果であり、最も重要なのは**「必要な成果物が、正しい順序で定義されているか」**です。

## 3. 定義すべき成果物（Deliverables）一覧

現状のベースライン（スコープ・スケジュール）および全体方針に基づき、本プロジェクトで作成すべき成果物を定義します。これらは目的に応じて柔軟に定義し直すことが可能です。

### A. 経営基盤（守り・全体接続）
**目的:** 経営の実態を数値で可視化し、意思決定の「ものさし」を作る。

| ID | 成果物名 | 目的・要件 | 備考 |
| :--- | :--- | :--- | :--- |
| **A-1** | **実質PL (v1)** | 会計上の数字ではなく、**「事業の実態」が見える損益計算書**。<br>要件：SFF、コミュニティ、研修ごとの収支が見えること。 | ベースライン定義済 |
| **A-2** | **経営判断マップ**<br>(KPIツリー) | **「どの数字を動かせば利益が出るか」**の構造図。<br>要件：現場のアクション（Call数など）とPLが繋がっていること。 | ベースライン定義済 |
| **A-3** | **SFF事業収支シミュレーション**<br>**(★新規追加)** | SFFの価格設定や開催判断を行うための専用PL。<br>要件：松竹梅のシナリオで、いくらの値付けなら赤字にならないかが一目でわかること。 | **Vision Logic対応** |
| **A-4** | **全体接続ロードマップ**<br>**(★新規追加)** | 全事業（点）がどう繋がり、どういう順序で進むかを示す「線」の図。<br>要件：依存関係が可視化されていること。 | **統合管理用** |

### B. 組織基盤（組織）
**目的:** 誰が何に責任を持つかを明確にし、タスクの滞留と不安を解消する。

| ID | 成果物名 | 目的・要件 | 備考 |
| :--- | :--- | :--- | :--- |
| **B-1** | **組織課題構造マップ** | 誰が何に困っているか（モヤモヤ）の可視化。<br>要件：固有名詞で課題の所在が特定されていること。 | ベースライン定義済 |
| **B-2** | **役割定義書 (RACI)** | **「誰が決めるのか（A）」「誰がやるのか（R）」**の明文化。<br>要件：ボールが落ちやすい領域（マーケと営業の間など）が埋まっていること。 | ベースライン定義済 |
| **B-3** | **「やらないこと」リスト**<br>**(★新規追加)** | リソース集中のための「捨てタスク」一覧。<br>要件：中井様の「思考の棚卸し」結果として、今は無視するものを明記する。 | **Mental Sorting対応** |

### C. 事業開発（攻め・IT）
**目的:** 仮説検証を最速で回し、収益化の確度を高める。

| ID | 成果物名 | 目的・要件 | 備考 |

| **C-2** | **法人研修マテリアル** | 実際に提供する研修コンテンツ。<br>要件：MVP（実用に足る最小限の機能）レベルで作成すること。 | ベースライン定義済 |
| **C-3** | **業務プロセスフロー (To-Be)** | ITツールを入れる前の**「理想の業務の流れ」**図。<br>要件：Salesforce導入前に、アナログで回るフローが完成していること。 | ベースライン定義済<br>(IT導入の前程) |

## 4. 具体的な役割分担（補完関係の構築）

中井様ご自身が「理念が先行し、意味をつなげることを大事にする」タイプであると自己分析されています。この特性を最大限活かすため、以下のような補完関係を築きます。

| | 役割 | 中井様 (COO) | 鈴木様 (No.2 / 参謀) |
| :--- | :--- | :--- | :--- |
| | **得意領域** | **理念・意味付け・熱量・対人折衝** | **構造化・数値根拠（PL）・実装・IT戦略** |
| | **アクション** | モヤモヤ（不安・課題・構想）を全てリストアップし、吐き出す。 | 吐き出された内容を「課題のリスト化」と「優先順位付け」を行い、解決策（ロードマップ・組織図）に落とし込む。<br>**※PL作成・管理の実務を巻き取る。** |
| | **ゴール** | メンバーへ想いを伝え、チームを鼓舞する。 | 「この通り進めば大丈夫」という安心感と論理的裏付けを提供する。 |

## 5. 現状の構造的課題と解決の方向性

音声ミーティングで顕在化した具体的な「詰まり」に対し、以下のアプローチを取ります。

*   **組織マネジメントの不全（タスクの滞留）**
    *   **課題:** 香音（Kaon）様が責任感から抱え込み、自信を喪失しやすい。的場（Matoba）様（マーケ）との連携が受動的。香織（Kaori）様のリソース配分（事務偏重）への懸念。神保（Jimbor）様・町田（Machida）様へのタスク委譲不全。
    *   **対策:** 「役割定義」と「権限委譲のライン」を明確化した組織図を再設計する。特に香織様については、強みである**コーチング領域（法人研修・社内ケア）へのシフト**を検討する。
*   **IT/CRM導入の順序**
    *   **課題:** Salesforce導入などの手段が目的化し、事業（SFF、コミュニティ）の要件定義前に進もうとしている懸念。
    *   **対策:** ロードマップ上で「事業戦略決定（価格・商品設計）」→「業務フロー整理」→「ツール導入」の順序を厳守させる壁打ちを行う。

## 6. 修正後のネクストアクション（進行ステップ）

前回設定した「Step 0」を、より具体的な**「デトックス＆構造化ミーティング」**へと昇華させます。

**Step 0：不安の棚卸しと接続ミーティング（最重要）**
*   **準備（中井様）:** ご自身の中で起きているモヤモヤ、課題、つながりが見えない部分を「リスト化・書き出し」していただきます。
*   **実施内容（鈴木様 - ヒアリングの観点）:**
    1.  **全出し:** 「SFFの価格設定の迷い」「香音様や的場様との連携の悩み」「PL運用の実態」など、固有名詞ベースで吐き出していただく。
    2.  **仕分け:** 「今解決すべきこと」と「後回し・無視して良いこと」をその場でジャッジする（**成果物B-3**の作成）。
    3.  **接続:** バラバラに見える課題が、全体戦略の中でどう位置づくかを解説・構造化する（**成果物A-4**の骨子作成）。

**Step 1：根拠あるロードマップと組織図の策定**
*   鈴木様が持ち帰り作業で**「数字的根拠（PL）」「実行体制（組織図）」「時間軸（ロードマップ）」**が整合した計画書を作成します。
*   **SFF事業のマイルストーン:** 集客、価格決定、開催時期の決定。
*   **法人研修のマイルストーン:** 2026年3月の全社導入判断に向けた経営者トライアルのスケジュール。

**Step 2：IT/CRM（Salesforce）の実装設計**
*   全体戦略（ロードマップ）が決まった後で、それを実現するためのツールとしてSalesforce等の要件定義を行います。

# 鈴木様へのアドバイス

次回のミーティングでは、何かを決めることよりも**「中井様の不安を受け止め、交通整理すること」**自体に最大の価値があります。

「リスト化してもらったものを一つ一つ潰す」というスタンスで臨み、
*   「SFFの価格設定は、この**PLシミュレーション（成果物A-3）**ができれば判断できますよ」
*   「香音様の負担は、**組織図（成果物B-2）**のこの役割を神保様に渡せば軽くなりますよ」
*   「Salesforceは、コミュニティのこの**業務フロー（成果物C-3）**が決まってから設定しましょう」

といったように、**バラバラの事象を「定義された成果物」に紐づけて、ひとつのストーリーにつなげてあげること**が、今回求められている最大の役割です。
