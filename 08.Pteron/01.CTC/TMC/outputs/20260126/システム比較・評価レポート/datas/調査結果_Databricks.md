# Databricks 調査結果

**調査日：** 2026年02月09日

---

## ⚠️ 重要な前提事項

### 調査対象について
- **「Databricks（vista）」というサービスは存在しません**
- **「vista」**はトヨタ自動車が構築したデータ統合基盤の名称であり、その構築にDatabricksプラットフォームが採用されています
- 本調査対象は**Databricks**（データ・インテリジェンス・プラットフォーム）となります

### 用途適合性の評価

**Databricksは以下の用途に特化したプラットフォームです：**
- ビッグデータ分析
- データレイクハウス構築
- AI/機械学習モデル開発・運用
- データエンジニアリング
- データサイエンス

**❌ 本案件の要件との適合性：不適合**

ご要望の以下の用途には、Databricksは**適していません**：
- 案件管理システム
- 進捗ステータス管理
- 入力フォーム（ITリテラシーが低いユーザー向け）
- 帳票出力（PDF見積書・報告書等）
- 承認ワークフロー
- 外部ユーザー（4,500名）向けポータル

**✅ 推奨される代替製品：**
- **Microsoft Power Platform**（PowerApps + Power Automate + SharePoint）
- **Salesforce**（Experience Cloud含む）
- **kintone**
- **ServiceNow**
- **Dynamics 365**

以下、参考情報としてDatabricksの詳細を記載します。

## B. ライセンス・料金体系

### プラン一覧

Databricksは**従量課金制（DBU: Databricks Unit）**を採用しており、エディションやワークロードタイプによって単価が異なります。

| プラン/エディション名 | 対象 | DBU単価（参考） | 年額単価 | 備考 |
|---------------------|------|---------|---------|------|
| Standard | 基本機能 | - | - | 小規模チーム向け |
| Premium | 高度なセキュリティ・ガバナンス | $0.55/DBU（汎用）<br>$0.30/DBU（ジョブ） | - | 企業向け推奨 |
| Enterprise | エンタープライズ機能フル | - | - | 要見積 |

**ワークロードタイプ別料金（Premium tier参考）：**
- All-Purpose Compute（対話的分析）：$0.55/DBU-hour
- Jobs Compute（バッチ処理）：$0.30/DBU-hour
- Jobs Light Compute：$0.15/DBU-hour
- SQL Warehouse：別途料金体系

**注意：** DBU消費量は使用するコンピュートリソース、処理データ量、クラスター構成により変動します。

### 本利用規模でのコスト概算

| 費用項目 | 金額（概算） | 算出根拠 |
|---------|-----------|---------|
| 初期費用 | 0円 | 従量課金制のため初期費用なし |
| 月額（社内ユーザー分） | **用途不適合** | Databricksは分析プラットフォームであり、案件管理・ワークフロー用途のユーザーライセンス体系は存在しない |
| 月額（社外ユーザー分） | **用途不適合** | 外部ユーザー4,500名が案件管理・入力フォームにアクセスする用途には設計されていない |
| 月額合計 | **算出不可** | 使用するコンピュート量とDBU消費により変動。分析・開発用途であれば数十万円〜数百万円/月 |
| ストレージ費用（月額） | クラウドプロバイダー依存 | AWS S3、Azure Blob、GCS等の料金（概算：数万円〜数十万円/月） |
| 5年間総コスト | **算出不可** | 利用パターンにより大きく変動 |

### 注意事項
- **Databricksは分析・AI開発プラットフォーム**であり、業務アプリケーション（案件管理、ワークフロー、外部ユーザーポータル等）の構築には適していません
- ユーザー数ではなく**コンピュート時間（DBU）**で課金されるため、従来型のユーザーライセンスとは異なる
- Databricks Appsを使用した簡易アプリケーション開発は可能ですが、複雑なワークフロー・承認フロー・帳票出力等には不向き
- 実際のコストはワークロード、データ量、クラスター構成、稼働時間により大きく変動

## C. 機能対応状況

| # | 機能 | 対応レベル | 実現方法 | 備考 |
|---|------|----------|---------|------|
| 1 | 統一ID管理 | **対応困難** | Delta Lake上でテーブル設計すれば技術的には可能 | 分析用途向けであり、トランザクション管理には不向き |
| 2 | ステータス管理 | **対応困難** | カスタムテーブル設計とNotebook/SQL | リアルタイムな業務システム向けではない |
| 3 | 入力フォーム | **追加開発が必要** | Databricks Apps（Streamlit/Dash等）でUI構築 | ITリテラシーが低いユーザー向けには複雑。スマホ対応は限定的 |
| 4 | ファイル添付・ストレージ | **標準機能** | Unity Catalog + クラウドストレージ（S3/ADLS/GCS） | データ分析用途には最適だが、業務ファイル管理には不向き |
| 5 | 帳票・出力 | **追加開発が必要** | PythonライブラリでPDF生成（ReportLab等） | テンプレートベースの帳票出力には大規模カスタム開発が必要 |
| 6 | 承認ワークフロー | **対応困難** | Databricks Workflowsは**データパイプライン用**であり業務承認フロー向けではない | 多段階承認・差し戻し等の業務ワークフローは実装困難 |
| 7 | 通知・アラート | **カスタマイズ** | Webhooks、Email API統合で実装可能 | ジョブ完了通知は標準機能。業務通知には追加開発が必要 |
| 8 | ダッシュボード・KPI | **標準機能** | Databricks SQL、Lakeview、AI/BI Genie | **これはDatabricksの得意分野** |
| 9 | アカウント・権限管理 | **標準機能** | Unity Catalog、RBAC、SSO対応 | データアクセス制御には優れているが、業務アプリの権限管理とは異なる |
| 10 | 監査ログ | **標準機能** | System Tables、Unity Catalog Audit Logs | データアクセス・操作履歴の記録は標準機能 |

### 各機能の詳細

**1. 統一ID管理**
- Delta Lakeテーブルで案件IDを管理することは技術的に可能
- ただし、リアルタイムなトランザクション処理や同時編集制御には不向き
- OLAPワークロード（分析）には最適だが、OLTPワークロード（業務システム）には不適

**2. ステータス管理**
- カスタムテーブルとSQL/Notebookで実装可能
- リアルタイムな状態管理や複雑な業務ロジックには追加開発が大規模に必要

**3. 入力フォーム**
- **Databricks Apps**を使用してStreamlit、Dash、Gradio等のフレームワークでUI構築可能
- ただし、ITリテラシーが低いユーザー向けの直感的なフォームUIには不向き
- スマートフォン対応は限定的（レスポンシブデザインは開発者が実装する必要あり）

**4. ファイル添付・ストレージ**
- Unity Catalogとクラウドストレージ（S3、Azure Blob、GCS）で大容量ファイル管理が可能
- データ分析用途には最適だが、業務ファイル管理（バージョン管理、承認履歴等）には追加実装が必要

**5. 帳票・出力**
- PythonライブラリでPDF生成は可能（ReportLab、fpdf等）
- ただし、テンプレートベースの帳票出力システム（見積書、報告書、保証書等）を構築するには大規模なカスタム開発が必要
- 標準的な帳票ツールと比較して開発工数が大幅に増加

**6. 承認ワークフロー**
- **Databricks Workflows**はデータパイプラインのオーケストレーション用であり、業務承認フローには使用できない
- 多段階承認、差し戻し、再申請などの業務ワークフローを実装するには、別のワークフローエンジンが必要

**7. 通知・アラート**
- ジョブ完了通知は標準機能
- 業務通知（ステータス変更、期限超過等）はWebhooks、Email APIを統合してカスタム実装が必要

**8. ダッシュボード・KPI**
- **これはDatabricksの得意分野**
- Databricks SQL、Lakeview、AI/BI Genieで高度な分析ダッシュボードが構築可能
- ただし、業務システムのリアルタイムKPIダッシュボードとは設計思想が異なる

**9. アカウント・権限管理**
- Unity Catalogによる統一的なデータガバナンス
- RBAC（ロールベースアクセス制御）対応
- SSO対応（SAML、OAuth）
- ただし、業務アプリケーションの組織階層・店舗別権限管理とは設計が異なる

**10. 監査ログ**
- System Tables、Unity Catalog Audit Logsで操作履歴を記録
- データアクセス・変更履歴の追跡は標準機能

## D. 外部ユーザー対応

| 項目 | 内容 |
|------|------|
| アクセス方式 | **用途不適合**：Databricksは分析プラットフォームであり、外部ユーザー向けポータルの構築には適していない |
| アカウント発行・管理 | Databricksアカウントは最大10,000ユーザーまで。外部ユーザー4,500名の管理は技術的には可能だが、コスト・運用面で非効率 |
| データアクセス制限 | Unity Catalogで細かいアクセス制御は可能。ただし、店舗別・企業別のデータ分離は大規模なテーブル設計が必要 |
| スケーラビリティ（4,500名） | コンピュート面ではスケール可能。ただし、同時アクセスユーザー数が多い業務システムとしての利用は想定されていない |
| 認証方式 | SSO（SAML）、OAuth、SCIMによるユーザープロビジョニング対応。多要素認証はIDプロバイダー側で設定 |
| ライセンス費用 | **用途不適合**：外部ユーザーが案件管理・入力フォームにアクセスする用途には設計されていない。コンピュート時間（DBU）で課金されるため、多数の外部ユーザーが頻繁にアクセスする用途ではコストが高騰するリスクあり |

### 重要な注意事項
- Databricksは**データサイエンティスト・データエンジニア向け**のプラットフォーム
- 外部ユーザー4,500名が案件管理・入力・承認等の業務を行う用途には**全く適していません**
- このような用途にはPower Apps、Salesforce Experience Cloud、kintone等の業務アプリケーションプラットフォームを推奨

## I. 総括

### 強み
1. **ビッグデータ処理**：ペタバイト級のデータを高速処理
2. **統合プラットフォーム**：データエンジニアリング、データサイエンス、BIを一つのプラットフォームで提供
3. **AI/ML機能**：エンドツーエンドの機械学習ライフサイクル管理
4. **オープンアーキテクチャ**：Apache Spark、Delta Lakeなどのオープンソース技術ベース
5. **エンタープライズ対応**：高度なセキュリティ、ガバナンス、コンプライアンス
6. **スケーラビリティ**：データ量・処理量の増加に柔軟に対応
7. **豊富な導入実績**：Fortune 500企業の60%以上、日本でも大手企業の導入多数

### 弱み・懸念

#### **❌ 本案件要件との不適合（重大）**
1. **用途の根本的な不一致**：
   - Databricksはデータ分析・AI開発プラットフォームであり、**業務アプリケーション基盤ではない**
   - 案件管理、ワークフロー、入力フォーム、帳票出力等の機能は標準提供されていない
   - これらを実装するには大規模なカスタム開発が必要

2. **外部ユーザー対応**：
   - 4,500名の外部ユーザー（店舗・企業）が案件管理・入力・承認を行う用途には全く適していない
   - コンピュート時間課金のため、多数のユーザーが頻繁にアクセスするとコストが高騰

3. **UI/UX**：
   - データサイエンティスト・エンジニア向けのUIであり、ITリテラシーが低いユーザー向けではない
   - スマートフォン対応は限定的

4. **承認ワークフロー**：
   - Databricks Workflowsはデータパイプライン用であり、業務承認フローには使用不可
   - 多段階承認、差し戻し等の実装は事実上困難

5. **帳票出力**：
   - テンプレートベースのPDF帳票出力システムを構築するには大規模開発が必要
   - 既製の帳票ツールと比較して開発工数が大幅に増加

#### その他の弱み
6. **学習コスト**：データエンジニアリング・プログラミングスキルが必要
7. **料金の複雑性**：DBUベースの従量課金で、コスト予測が難しい
8. **オーバースペック**：分析・AI用途以外では過剰な機能

## **最終結論・推奨事項**

### ❌ **Databricksは本案件要件に不適合**

**理由：**
1. Databricksはデータ分析・AI開発プラットフォームであり、業務アプリケーション基盤ではない
2. 案件管理、ワークフロー、入力フォーム、帳票出力等の機能は標準提供されていない
3. 外部ユーザー4,500名が利用する業務システムとしての設計思想がない
4. 実装には大規模なカスタム開発が必要で、コスト・工数・リスクが非常に高い

### ✅ **推奨される代替製品**

| 製品 | 適合度 | 推奨理由 |
|------|-------|---------|
| **Microsoft Power Platform** | ⭐⭐⭐⭐⭐ | PowerApps（フォーム・UI）+ Power Automate（ワークフロー）+ SharePoint（ファイル管理）の統合。外部ユーザー対応あり |
| **Salesforce** | ⭐⭐⭐⭐⭐ | Experience Cloud（旧Community Cloud）で外部ユーザーポータル構築可能。ワークフロー・承認機能が充実 |
| **kintone** | ⭐⭐⭐⭐ | 日本企業に最適化されたローコード開発。直感的なUI。外部ユーザー対応あり |
| **ServiceNow** | ⭐⭐⭐⭐ | エンタープライズ向け業務管理プラットフォーム。ワークフロー・承認機能が強力 |
| **Dynamics 365** | ⭐⭐⭐⭐ | Microsoft製品との統合が強力。外部ポータル対応 |

- ビッグデータ分析
- データレイクハウス構築
- 機械学習モデルの開発・運用

- 予測分析・レコメンデーション
