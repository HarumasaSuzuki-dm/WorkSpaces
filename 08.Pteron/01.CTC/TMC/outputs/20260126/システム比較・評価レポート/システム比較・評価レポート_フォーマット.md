## 2. 本レポートの目的と前提条件

### 2.1 目的

本レポートは、TMC様における充電器販売・施工業務のDX推進に向けて、最適なシステム基盤を選定するための比較・評価資料である。

2026年2月の「選定意思決定」マイルストーンに向け、客観的な比較情報を提供し、意思決定を支援することを目的とする。

### 2.2 前提条件

| 項目 | 内容 |
|------|------|
| 対象業務 | BEV/PHEV充電器の販売・施工に関するアフターサービス業務 |
| 利用者 | TMC、車両販売店（約4,000店）、施工管理会社、施工会社 |
| 想定データ量 | 【ヒアリングで確認】 |
| セキュリティ要件 | 個人情報（顧客名、住所等）を含むため、高いセキュリティレベルが必要 |
| 導入希望時期 | FY26（2026年4月以降） |

### 2.3 本レポートのスコープ

**スコープ内：**
- システム基盤の比較・評価
- 導入に向けた概算コスト・期間の整理

**スコープ外：**

## 3. TMC様の現状課題の整理

### 3.1 課題一覧

2026年1月21日の週次定例にてTMC様よりご共有いただいた課題を整理する。

| No. | 課題カテゴリ | 具体的な課題内容 | 影響 |
|-----|----------|-------------|------|
| 1 | 進捗把握 | 工事がいつ終わるかわからない | フォロー業務の遅延 |
| 2 | 進捗把握 | 工事予定がわからずフォローできない | リマインドのタイミングが不明 |
| 3 | 情報収集 | 受注台数を把握しにくい | 実態把握困難 |
| 4 | フォロー負荷 | 「出してください」という催促がしんどい | 工数増加 |
| 5 | 情報経路 | 通過する工程が多い（伝言ゲーム状態） | 情報の遅延・欠落 |
| 6 | 受領手段 | 各社からメールやTeams上で受領（手段がバラバラ） | 管理工数増加 |
| 7 | ID管理 | 共通のID管理ができない | 各社ごとの個別管理が必要 |
| 8 | フォーマット | 各社ごとに書類フォーマット、記載内容がバラバラ | 品質評価困難 |
| 9 | データアクセス | 市場不具合発生時にすぐにアクセスしたい | 対応遅延リスク |
| 10 | 自動化 | PADで転記自動化を試みたが、フォーマット・形式がバラバラで苦戦 | 手作業が残存 |

### 3.2 課題の構造化

```
【根本課題】
情報の型が決まっていない／共通の型を持っていない

    ├─【派生課題1】フォーマットがバラバラ
    │    └─ 品質評価困難、自動化困難
    │
    ├─【派生課題2】共通ID管理ができない
    │    └─ 各社ごとの個別管理、突合困難
    │
    ├─【派生課題3】受領手段がバラバラ
    │    └─ 情報散在、管理工数増加
    │
    └─【派生課題4】進捗が見えない
         └─ フォロー困難、リマインド非効率
```

### 3.3 システム化により解決すべき要件

| 優先度 | 要件 | 対応する課題No. |
|--------|------|---------------|
| 必須 | 共通の管理IDで案件を一元管理できること | 7 |
| 必須 | 工事の進捗ステータスを一覧で確認できること | 1, 2 |
| 必須 | 個人情報を安全に管理できること | （セキュリティ要件） |
| 高 | 統一フォーマットで情報を入力・出力できること | 8, 10 |
| 高 | デジタルリテラシーの低いユーザーでも使いやすいこと | （運用要件） |
| 中 | 既存システム（ai21等）と連携できること | （拡張性要件） |
| 中 | 不具合情報に迅速にアクセスできること | 9 |

## 4. 比較対象システムの概要

### 4.1 比較対象一覧

| 候補 | システム/方式 | 概要 |
|------|-------------|------|
| A | vista（Databricks）連携型 | TMC社内データ基盤と連携した構築 |
| B | クラウドSaaS型 | 既製のクラウドサービスを活用 |
| C | スクラッチ開発 | AWS/Azure上に新規構築 |
| D | 既存システム拡張 | SharePoint + Power Automate強化 |

※ 候補はヒアリング結果に基づき追加・変更する可能性があります。

---

### 4.2 候補A：vista（Databricks）連携型

#### 概要

TMC様が社内で構築を進めているデータ基盤「vista」と連携し、充電器販売・施工業務のデータを一元管理する方式。

#### 特徴

| 項目 | 内容 |
|------|------|
| 基盤 | Databricks（Apache Sparkベース） |
| 運用主体 | TMC社内 |
| 特徴 | 社内データとの統合が容易、将来的なAI活用の土台 |

#### TMC様環境での適用イメージ

【ヒアリング後に記載】

#### メリット

- 社内データ基盤との親和性が高い
- 将来的なデータ分析・AI活用への拡張が容易
- 社内ガバナンスの範囲内で管理可能

#### デメリット・留意点

- 入力フォーム部分は別途開発が必要
- 施工会社等の外部ユーザーへのアクセス方法の検討が必要

---

### 4.3 候補B：クラウドSaaS型

#### 概要

市販のクラウドサービス（例：Salesforce、Kintone、Microsoft Power Platform等）を活用し、短期間でシステムを構築する方式。

#### 特徴

| 項目 | 内容 |
|------|------|
| 基盤 | SaaSベンダー提供のクラウド環境 |
| 運用主体 | ベンダー + TMC |
| 特徴 | 短期間で導入可能、ノーコード/ローコードで柔軟にカスタマイズ |

#### TMC様環境での適用イメージ

【ヒアリング後に記載】

#### メリット

- 導入が比較的早い
- 標準機能が豊富でカスタマイズしやすい
- 運用負荷が軽い

#### デメリット・留意点

- セキュリティ審査の対象となる
- 社内データ基盤との連携に追加開発が必要な場合がある
- ランニングコストが継続的に発生

---

### 4.4 候補C：スクラッチ開発

#### 概要

AWS/Azure等のクラウドインフラ上に、TMC様の要件に合わせたシステムを一から構築する方式。

#### 特徴

| 項目 | 内容 |
|------|------|
| 基盤 | AWS / Azure / GCP |
| 運用主体 | TMC + 開発ベンダー |
| 特徴 | 要件に完全に合わせた設計が可能 |

#### TMC様環境での適用イメージ

【ヒアリング後に記載】

#### メリット

- 要件に100%合わせた設計が可能
- 将来的な機能拡張の自由度が高い

#### デメリット・留意点

- 開発期間・コストが大きい
- セキュリティ審査（公開Webサイト、クラウド）が必要
- 運用・保守の体制確保が必要

---

### 4.5 候補D：既存システム拡張

#### 概要

現在TMC様が利用しているSharePoint + Power Automate（PAD）の仕組みを強化し、対応範囲を広げる方式。

#### 特徴

| 項目 | 内容 |
|------|------|
| 基盤 | Microsoft 365（SharePoint、Power Automate） |
| 運用主体 | TMC |
| 特徴 | 既存資産の活用、追加コストを抑制 |

#### TMC様環境での適用イメージ

【ヒアリング後に記載】

#### メリット

- 既存環境の延長で導入可能
- 追加のセキュリティ審査が不要な可能性
- 初期コストを抑えられる

#### デメリット・留意点

- 各社フォーマットがバラバラな課題は根本解決にならない
- 外部ユーザー（施工会社等）への展開に制約
- 機能拡張に限界がある

## 5. 評価軸の定義

### 5.1 評価軸一覧

| No. | 評価軸 | 定義 | 重み |
|-----|--------|------|------|
| 1 | 統一ID管理 | 共通の案件IDで全関係者が同じ情報を参照できるか | 高 |
| 2 | 入力のしやすさ | デジタルリテラシーが低いユーザーでも迷わず入力できるか | 高 |
| 3 | フォーマット統一 | 入力・出力のフォーマットを統一できるか | 高 |
| 4 | ステータス管理 | 工事の進捗を一覧で把握し、フォローに活用できるか | 高 |
| 5 | セキュリティ | 個人情報を含むデータを安全に管理できるか | 必須 |
| 6 | 既存システム連携 | ai21、TQ-NET、vista等との連携が可能か | 中 |
| 7 | コスト | 初期費用・ランニングコストは妥当か | 中 |
| 8 | 導入リードタイム | セキュリティ審査を含め、いつから使い始められるか | 中 |

### 5.2 評価基準

| 評価 | 基準 |
|------|------|
| ◎ | 要件を十分に満たし、追加の対応なしで実現可能 |
| ○ | 要件を満たすが、一部カスタマイズや運用での対応が必要 |
| △ | 要件を満たすことは可能だが、大きな制約や追加コストが発生 |
| × | 要件を満たすことが困難、または対応不可 |
